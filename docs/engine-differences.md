# Search Engine Differences - Operator Support and Query Syntax

## Why Engine Choice Matters for Dorking

Google, Bing, and DuckDuckGo have fundamentally different query parsers. A query that returns 200 precise results on Google may return zero results on Bing if it uses nested parentheses in the wrong position. DuckDuckGo may silently ignore the Boolean structure entirely.

OnlyNerds generates a **separate query for each engine** based on documented parser behavior. The engine selector is Step 1 in the Job Search Builder because the engine choice changes how every other input is encoded.

---

## Operator Support Matrix

| Operator | Google | Bing | DuckDuckGo |
| -------- | ------ | ---- | ---------- |
| `site:` | Full | Max 3, flat preferred | Single site only |
| `"exact phrase"` | Full | Full | Full |
| `OR` (uppercase) | Required | Required (strict case) | Partial |
| `( )` grouping | Full nested | Flat only | Ignored / flattened |
| `-exclusion` | Standard | Standard | Aggressive (over-excludes) |
| `intitle:` | Full | Unreliable | Not supported |
| `inurl:` | Full | Unreliable | Not supported |
| `tbs=` date filter | Yes (URL param) | No | No |
| `filetype:` | Full | Partial | Not supported |

---

## Google

### Strengths

- Full nested Boolean: `(A OR B) AND (C OR D)` works exactly as written
- Unlimited `site:` operators in an OR chain
- `tbs=` URL parameter for reliable date filtering (past hour, day, week)
- Fastest ATS indexing - new postings appear within 24-48 hours
- Full `intitle:` and `inurl:` support

### Weaknesses

- Strictest IP-level rate limiting of the three
- Complex queries from shared IPs trigger CAPTCHAs faster
- Requires jitter token to reduce shared-IP detection

### Query structure (generated by builder)

```
(role terms OR aliases)
(level terms)
(location terms)
(site:ats1.com OR site:ats2.com OR site:ats3.com)
"company name"
```

Each group is on its own line in the preview, joined with implicit AND. Every OR must be uppercase (though Google is tolerant of lowercase `or`, it's still uppercase by convention).

### Date filter

Appended to the URL, not the query string:
- `tbs=qdr:h` - past hour
- `tbs=qdr:d` - past day
- `tbs=qdr:w` - past week
- `tbs=qdr:m` - past month

## Bing

### Strengths

- Much more lenient on complex queries from shared IPs
- No meaningful rate-limiting for normal search behavior
- Indexes Greenhouse, Lever, Workday, and Ashby reliably
- Boolean OR works well when queries are kept flat

### Weaknesses

- `site:` operators placed inside nested parentheses are often silently dropped beyond the 3rd
- `OR` must be uppercase - lowercase `or` is treated as a literal search keyword
- Alias expansion must be limited (long OR chains reduce precision on Bing)
- Level terms with complex nested OR blocks often get ignored; primary keyword is more reliable
- No `tbs=` - use Bing's "Date" dropdown in the results UI instead

### Query structure (generated by builder)

```
site:ats1.com OR site:ats2.com OR site:ats3.com
"role" OR "alias1" OR "alias2"
"primary level term"
"city" OR "region"
"company name"
```

Key differences from Google:
- **Sites come first** - Bing handles site: operators better when they appear at the start of the query
- **No outer parentheses on the site block** - Bing's OR parsing on `site:` works better flat
- **Max 3 sites total** - the builder enforces this across all selected ATS tiers
- **2 aliases per role maximum** - keeps the OR chain short enough for Bing to parse cleanly
- **Level simplified** - extracts only the first quoted term from each level's query string

### Bing-specific behavior notes

**OR case sensitivity:** Bing is strictly case-sensitive about `OR`. If you copy a Bing query and manually edit it to use lowercase `or`, those terms will be treated as keyword matches, not Boolean operators.

**Parentheses:** Bing can handle one level of parentheses but often ignores inner nesting. `(A OR B) AND (C OR D)` usually works. `(A OR (B OR C))` often doesn't. The builder avoids inner nesting entirely for Bing.

**Jitter:** No jitter token is added on Bing. The minus operator on Bing works at a higher level than Google's - adding a minus term can affect entire page categories, not just string matches. Since Bing also has no meaningful shared-IP rate limiting, jitter is unnecessary.

## DuckDuckGo

### Strengths

- Does not log IP addresses or search history (strongest privacy posture)
- Aggregates from 400+ sources primarily via Bing's API - solid ATS coverage
- No IP rate limiting for normal search behavior
- Ideal for users on sensitive networks or who want zero search history

### Weaknesses

- Cannot handle nested parentheses - they are flattened into keyword searches
- OR chains with many terms are often ignored (DDG treats them as keywords)
- Only a single `site:` operator is reliable per query
- Does not support `intitle:`, `inurl:`, or `filetype:`
- Date filtering only available via UI dropdown in results - not in the URL
- If a query is too complex, DDG returns zero results with no error message

### Query structure (generated by builder)

```
"primary role" "optional second role" "first level term" "location" site:first-ats.com "company"
```

Everything on a single line, no parentheses, no OR chains. The builder enforces these constraints:
- Maximum 2 roles (primary labels only, no aliases)
- Maximum 1 level term (first selected level, primary quoted term only)
- Maximum 1 location (custom city takes priority over hub selection)
- Maximum 1 ATS site (first site from first selected tier)

### The "silent failure" problem

DDG does not show a CAPTCHA. When it decides a query is too complex or returns no results, it shows a "We couldn't find anything" page. This is indistinguishable from a legitimate zero-result search.

If you're getting zero results on DDG:
1. The query may be too complex (even with the builder's simplification)
2. Try selecting fewer inputs
3. Try Bing instead - same ATS indexing, more tolerant parser

### Bangs vs dorks

DDG is designed around "bangs" (`!g` for Google, `!so` for Stack Overflow). Complex Boolean dorks are not its native use case. The builder uses DDG as a privacy-safe fallback, not as a primary dorking engine.

## Choosing the Right Engine

```
Do you need: full alias expansion + multiple ATS sites + date filter?
  → Google (be aware of cooldown)

Are you on a shared network (dorm, office, library)?
  → Bing (more lenient, good ATS coverage, no cooldown)

Do you want zero IP logging or are you on a flagged network?
  → DuckDuckGo (simplified query, but maximum privacy)

Is Google showing you CAPTCHAs?
  → Switch to Bing immediately. Google CAPTCHA clears in 4-24 hours.
  → DuckDuckGo if Bing also has issues.
```

## Per-Engine Query Builder Functions

These are defined in `src/components/jobsearch/JobSearchBuilder.tsx`:

| Function | Engine | Notes |
| -------- | ------ | ----- |
| `buildGoogleQuery()` | Google | Full nested Boolean, all aliases, all sites |
| `buildBingQuery()` | Bing | Sites first, flat OR, max 3 sites, 2 aliases/role |
| `buildDDGQuery()` | DuckDuckGo | Single line, no OR chains, 1 site, 2 roles max |

The engine URL is then built by `buildEngineUrl()` in `src/data/dorkSafety.ts`, which applies jitter to Google queries only.
